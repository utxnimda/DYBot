# 项目 AI 协作规范

本文档约束所有参与本项目的 AI coding agent 和人工协作者。任何代码改动开始前，必须先阅读本文和指定的基础规范。

## 强制阅读清单

在进行任何代码改动、目录调整、依赖新增、构建脚本修改前，必须阅读：

1. [工程基础要求](engineering-foundation.md)
2. [项目目录结构规范](project-structure.md)
3. [架构提案](desktop-bot-assistant-architecture.md)
4. [ADR 0001：Electron + TypeScript + Vue/Vite](adr/0001-electron-typescript-vue.md)
5. 当前任务涉及包的 `README.md` 或包内说明文档，如果存在。

如果任务涉及安全、IPC、密钥、AI prompt、TTS、音频播放、数据库 migration，必须重新检查对应章节，不允许凭记忆改。

## 开工前流程

每次开始代码改动前必须做：

- 确认当前任务目标。
- 阅读强制阅读清单。
- 检查当前工作树状态。
- 查找相关文件和既有模式。
- 判断是否需要新增 ADR。
- 判断是否需要更新文档、测试、配置样例。
- 明确本次改动影响的 package 边界。

不允许：

- 不读规范直接写代码。
- 未理解现有目录边界就新增目录。
- 未检查同类实现就引入新模式。
- 为了短期方便绕过 `packages/contracts`。

## 代码改动原则

必须遵守：

- 模块边界优先。
- 先 contract，再实现，再 UI。
- 外部输入先 schema 校验。
- renderer 不直接访问 Node、文件、数据库、socket、密钥。
- 任何跨进程通信走 typed IPC。
- 任何跨模块通信走 typed event/command。
- 所有错误归一成项目错误模型。
- 日志必须有 `traceId`，敏感字段必须脱敏。
- 新增 provider 必须通过 adapter。
- 新增配置必须有 schema、默认值和文档说明。

## 目录改动规则

任何目录改动必须符合 [项目目录结构规范](project-structure.md)。

根 `config/`：

- 只放工程配置。
- 不放用户配置。
- 不放密钥。

包内 `config/`：

- 只放包级默认定义、schema、presets、模板。
- 不放用户运行时配置。

运行时用户数据：

- 只放用户数据目录。
- 不进入仓库。

## 依赖新增规则

新增依赖前必须说明：

- 为什么标准库或现有依赖不够。
- 依赖维护状态。
- 是否影响 Electron 打包。
- 是否有原生模块。
- 是否影响自动更新包体。
- 是否有安全风险。

禁止：

- 随手引入重型库。
- 引入未维护库。
- 引入会绕过 Electron 安全边界的库。
- 在 renderer 中引入需要 Node 权限的库。

## 测试要求

改动必须匹配测试层级：

- 纯函数和协议解析：包内 unit test。
- 跨包 pipeline：`tests/integration/`。
- UI 行为：`tests/e2e/`。
- AI/TTS/斗鱼网络调用：默认 mock 或 fixture，不依赖真实 key。

不能运行测试时，必须在最终说明中明确原因和风险。

## 文档同步规则

以下情况必须更新文档：

- 目录结构变化。
- 新增 package。
- 新增配置项。
- 新增 provider。
- 修改 IPC contract。
- 修改数据库 schema/migration。
- 修改发布流程。
- 新增安全例外。

短期无法满足规范时，必须新增或更新 ADR，说明：

- 例外原因。
- 风险。
- 临时方案。
- 回补计划。

## 安全红线

禁止：

- 提交真实 API key、token、cookie、私钥。
- 在日志中输出密钥。
- renderer 保存或读取密钥。
- 加载远程页面作为主 UI。
- 使用 `eval` 或动态执行字符串。
- 让弹幕内容进入 system prompt。
- 让用户输入控制本地文件路径后直接读写。
- 让用户输入控制任意 URL 后由 main 直接请求。

## AI Prompt 规则

必须：

- 系统指令、人设、弹幕原文分离。
- 弹幕原文作为 data block。
- 输出默认短句，适合 TTS。
- 对不适合直播播放的文本做拒绝、改写或跳过。
- 记录模型、耗时、token 估算。

禁止：

- 把弹幕直接拼进 system prompt。
- 让弹幕里的指令覆盖项目规则。
- 输出密钥、内部配置、系统路径。

## 最终回复要求

完成任务后必须说明：

- 改了哪些文件。
- 为什么这样改。
- 运行了哪些验证。
- 哪些验证没跑及原因。
- 是否还有风险或后续工作。

如果只是文档改动，也要说明没有运行构建/测试的原因。
